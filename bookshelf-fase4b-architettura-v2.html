Dettaglio file

AuthContext.jsx
AuthProvider
Gestisce utente, erroreLogin, inCaricamento. Al montaggio controlla se c'è già un token in localStorage con isAutenticato(). Chiama login() e logout() da api.js.
useAuth()
Hook che legge il valore corrente di AuthContext. Qualsiasi componente dentro AuthProvider può chiamarlo senza ricevere props.
Perché AuthProvider è esterno a LibreriaProvider?
LibreriaContext chiama useAuth() internamente per sapere quando l'utente fa login o logout e ricaricare i libri. Deve quindi trovarsi dentro AuthProvider.
LibreriaContext.jsx
LibreriaProvider
Chiama useAuth() per leggere utente. Quando utente cambia (useEffect), ricarica i libri dal server o svuota lo stato se l'utente è uscito.
useLibreria()
Hook che espone libreria, statistiche e tutte le funzioni CRUD. Usato direttamente nelle pagine senza passare props.

Schema — database bookshelf

tabella · utenti
utenti
id PKINT AUTO_INCREMENT
usernameVARCHAR(50) UNIQUE
password_hashVARCHAR(255)
nomeVARCHAR(100)
La password non viene mai salvata in chiaro. bcrypt.hash() produce un hash diverso ogni volta anche con la stessa password. Il confronto avviene con bcrypt.compare().
tabella · libreria
libreria
id PKVARCHAR(100) — id Open Library
titoloVARCHAR(255)
autoreVARCHAR(255)
annoVARCHAR(10)
copertinaVARCHAR(500) NULL
statoENUM(da-leggere, in-lettura, letto)
utente_id FKINT → utenti.id
PRIMARY KEY (id, utente_id) — chiave composta. Lo stesso libro può stare nella libreria di utenti diversi.

ON DELETE CASCADE — se un utente viene eliminato, tutti i suoi libri vengono eliminati automaticamente.

Script

db-init.sql
Crea il database bookshelf, le tabelle utenti e libreria, e inserisce l'utente di prova con password già hashata. Da eseguire una sola volta in MySQL Workbench.
crea-utenti.js
Script Node per aggiungere utenti senza SQL manuale. Modifica l'array nuoviUtenti e lancia con node crea-utenti.js. Se lo username esiste già lo salta senza crashare.
◈ Flusso autenticazione
1
L'utente inserisce username e password in Login.jsx e clicca Accedi.
2
AppInterna chiama handleLogin() preso da useAuth().
3
AuthProvider chiama login() da api.js che fa POST /api/auth/login.
4
Il server verifica le credenziali con bcrypt, genera un JWT e lo restituisce.
5
api.js salva il token in localStorage. AuthProvider aggiorna utente nello stato.
6
LibreriaProvider rileva che utente è cambiato (useEffect) e carica i libri dal server.
7
AppInterna vede utente !== null e mostra le Routes al posto di Login.
→ Flusso ricerca libri
1
L'utente cerca un titolo in SearchBar. useRicerca() chiama cercaLibri() da api.js.
2
api.js fa GET /api/libri/cerca?q=termine al server (senza token — endpoint pubblico).
3
Il server chiama Open Library, normalizza i dati e li restituisce. Il client non conosce Open Library.
4
I risultati vengono mostrati in RisultatiRicerca. L'utente può aggiungere un libro.
⬡ Flusso aggiunta libro
1
L'utente clicca Aggiungi in BookCard. La pagina chiama aggiungiLibro() preso da useLibreria().
2
LibreriaProvider chiama aggiungiLibroApi() da api.js.
3
api.js fa POST /api/libreria con il token nell'header. Il server verifica il token con verificaToken.
4
Il controller inserisce il libro in MySQL legato all'utente_id estratto dal token.
5
Il server restituisce il libro inserito. LibreriaProvider aggiorna lo stato locale. Tutti i componenti che usano useLibreria() si aggiornano automaticamente.
◈ Come il Context evita il prop drilling
✗
Prima (prop drilling): App passava libreria, aggiungiLibro, rimuoviLibro, cambiaStato, isInLibreria a Home → Home le passava a BookCard.
✓
Ora (Context): Home chiama useLibreria() e prende solo quello che serve. BookCard potrebbe fare lo stesso direttamente, senza intermediari.
